# Rust Integration Guide: Google BigQuery

## Overview
Google BigQuery is an enterprise data warehouse solving analytics over petabytes of data using a serverless architecture. Writing to BigQuery seamlessly from Rust allows developers to create high-throughput data pipelines efficiently.

## APIs & Methods
Interacting with BigQuery in Rust revolves around GCP's official REST or gRPC APIs.

### 1. `gcloud-sdk` / `google-cloud-bigquery`
- **Crate:** `google-cloud-bigquery`
- **Focus:** Community-maintained crates utilizing asynchronous clients based on `hyper` and `reqwest` that interact directly with the BigQuery REST and gRPC endpoints.
- **Method:** `Client::insert_all`, `Storage API` (for high-throughput streaming).
- **Example Usage (REST Insert):**
  ```rust
  use google_cloud_bigquery::client::{Client, ClientConfig};
  use google_cloud_bigquery::http::tabledata::insert_all::{InsertAllRequest, Row};

  async fn stream_to_bigquery(client: &Client, project: &str, dataset: &str, table: &str, data: Row) {
      let mut req = InsertAllRequest::default();
      req.rows.push(data);
      let _ = client.tabledata().insert_all(project, dataset, table, &req).await;
  }
  ```

### 2. BigQuery Storage Write API (gRPC)
- **Crate:** `tonic`, generic Protobuf builds.
- **Focus:** Google recommends transitioning from the legacy `tabledata.insertAll` to the Storage Write API over gRPC for significant cost reduction and exactly-once semantics. Rust handles gRPC flawlessly via `tonic`.

## Configurations
- **Authentication:** `google-cloud-auth` crate seamlessly integrates Application Default Credentials (ADC) or explicit Google Service Account JSON files across async Runtimes.
- **Region:** BigQuery processing is regional; ensure your app points to the correct endpoint or dataset logical region (e.g., `EU` or `US-CENTRAL1`).

## Features & Transformations
- **Streaming Inserts:** Pushes rows individually or in micro-batches directly into the analytics engine. Data becomes available for querying almost instantly.
- **Staged Loads (Cloud Storage):** If throughput exceeds streaming quotas, Rust easily connects to `google-cloud-storage` using the Storage SDK, dumps `NDJSON` or `Parquet` files, and triggers a BigQuery Load Job asynchronously.
- **Parquet Support:** For maximal efficiency, serialize arrays of structs in Rust straight to Apache Parquet and trigger a GCS load job.

## Pros & Cons
### Pros
- **Serverless Scale:** Zero infrastructural limit on simultaneous connections from scaled Rust microservices.
- **JSON Compatibility:** BigQuery natively handles `JSON` or explicitly typed `NDJSON` structures corresponding perfectly to `serde_json` structs in Rust.

### Cons
- **Quota Limitations:** The legacy streaming API is subject to HTTP throughput quotas. If exceeded, latency spikes occur.
- **gRPC Complexity:** Implementing the newer Storage Write API requires generating Protobuf models locally via `prost` and `tonic-build`, heavily complicating build scripts (`build.rs`).

## Issues & Workarounds
- **Issue: Exactly-once streaming difficulties.**
  - *Workaround:* Using `insertAll` does not guarantee exactly-once delivery upon network retries unless `insertId` is explicitly provided for every row. Always explicitly bind an `insertId` using `uuid::Uuid::new_v4()` in Rust.
- **Issue: Complex JSON Schema Drifts.**
  - *Workaround:* If your Rust models evolve, BigQuery rejects streaming inserts for missing schemas. Configure BigQuery tables to tolerate loosely-typed JSON strings inside a specific `payload` column or ensure strict struct versioning locally.

## Recommendations & Suggestions
1. **Prefer the Storage Write API:** For extensive production loads, invest the time to build `tonic` stubs for the BigQuery Storage Write gRPC API. It halves API costs and exponentially increases throughput.
2. **Batching over HTTP:** If utilizing the legacy REST `insertAll`, never stream 1 row per POST request. Group them into arrays of 500-10,000 items locally within partitioned Tokio tasks to prevent running out of sockets.
3. **Partitioning:** Ensure the table you are sinking into is partitioned by a `Date`/`Timestamp` column natively generated by Rust (`chrono::Utc::now()`) to save BigQuery analysis costs later.
